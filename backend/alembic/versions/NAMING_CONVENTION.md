# Alembic Migration Naming Convention

## Standard Format

All migration files **must** follow this naming pattern:

```
{revision_id}_{short_description}.py
```

**Examples:**
```
6bce34dcf2df_add_document_and_kyc_document.py
24d552ed8ba9_add_audit_log.py
90bfd38432e5_add_partner_assignment_deliverable.py
```

## Rules

1. **Always include the revision hash prefix** — Alembic generates an 8–12 character hex ID when you run `alembic revision`. Keep it as the filename prefix. Files without a hash prefix (e.g. `add_brief_pdf_path.py`) are non-standard and should be avoided going forward.

2. **Use a short, lowercase, underscore-separated description** — The slug after the hash should describe *what changes*, not *why*. Aim for 3–6 words.
   - ✅ `add_partner_capability_matrix`
   - ✅ `add_escalation_and_sla_tracking`
   - ❌ `fix` / `update` / `misc_changes`

3. **One logical change per file** — Each migration should address a single feature or schema concern. If you need to add two unrelated tables, use two migrations.

4. **Never reuse a description** — Each slug must be unique. If you're revising a previous migration (e.g. adding a missing column), name it descriptively: `add_expiry_date_to_kyc_documents`, not `add_kyc_document` again.

## Generating a New Migration

```bash
# Auto-generate from model diff (then review and clean up the file)
cd backend && alembic revision --autogenerate -m "add_your_description_here"

# Create an empty migration for manual DDL
cd backend && alembic revision -m "add_your_description_here"
```

Alembic will name the file `{hash}_add_your_description_here.py` automatically.

## Avoiding Duplicates

Before creating a new migration, check the existing chain:

```bash
cd backend && alembic history
```

If you accidentally generate a duplicate (same tables/columns as an existing migration):
1. Identify which file is the canonical version (prefer the hand-written one with explicit `server_default` values and dialect-specific types over raw autogenerated output).
2. Delete the duplicate file.
3. Update the canonical file's `down_revision` to point directly to the predecessor of the deleted file.
4. Re-run `alembic history` to confirm the chain is intact.

> **Note:** The file `82526a45a2c4_add_document_and_kyc_document.py` was removed in March 2026 because it was an autogenerated duplicate of `6bce34dcf2df_add_document_and_kyc_document.py`. The hand-written version (`6bce34dcf2df`) was kept as it included proper `server_default` values and explicit `UUID(as_uuid=True)` types.
