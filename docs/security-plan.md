# AMG Portal Security & Encryption Plan

**Status:** Draft v1 — April 20, 2026
**Scope:** Full-platform encryption + defense-in-depth for AMG Portal (backend, frontend, mobile)
**Constraints:** US-only operations for now · free / OSS only, no paid services · AMG is itself an incident-response firm (no external IR retainer needed)
**Classification:** Internal — confidential

---

## 1. Executive summary

AMG Portal serves UHNW (ultra-high-net-worth) individuals, family offices, and global executives. Anchor Mill Group's programs include incident-response services, which means the platform holds **both** sensitive wealth data (KYC, financial records, private communications) **and** IR casework (forensics, breach notes, evidence, chain-of-custody material). Two categories, both bank-grade sensitive.

**Current posture:** authentication, rate limiting, HSTS, append-only audit table, and a nonce-generating CSP middleware all exist and are above average. **But** PII (tax IDs, passports, DOB, net worth, OAuth tokens, DocuSign private key) is stored plaintext; there is no CSRF protection under the cross-origin `SameSite=none` cookie setup; the CSP nonce generated by frontend middleware is silently thrown away because no layout consumes it; the audit log has no tamper-evidence; password hashing is still bcrypt.

**Critical architectural decisions:**

1. **Messaging is server-side envelope-encrypted with supervised break-glass, NOT true E2EE.** The platform must be able to supervise and retain advisor-client communications for its clients (regulatory future-proofing) and for IR case context.
2. **Key hierarchy is KEK-in-env-var with versioning + HKDF per-tenant DEKs.** Free, Railway-compatible, code-only. Designed for drop-in migration to self-hosted Vault OSS later if the operational footprint grows.
3. **Column encryption is hand-rolled SQLAlchemy `TypeDecorator` + AES-256-GCM with AAD + key-id header byte + HMAC blind indexes** for the handful of fields that need equality lookup. Not `sqlalchemy_utils.EncryptedType` (maintenance mode, breaks SQLA 2 caching, no AAD, no versioning).
4. **File vault is client-side envelope encryption on top of MinIO SSE-S3,** with ClamAV scanning of plaintext pre-encryption, object-lock in COMPLIANCE mode for evidentiary files, proxy-through downloads for sensitive files, short-TTL one-time-redemption presigned URLs only for large files, and crypto-shred (delete per-subject DEK) for GDPR-style erasure that doesn't break retention.
5. **Audit chain is tamper-evident:** per-row `prev_hash` + HMAC + daily Ed25519-signed Merkle root + RFC 3161 timestamp from FreeTSA.org. Non-app DB role revokes UPDATE/DELETE on the audit table. Daily verification cron alerts on chain break. Evidentially admissible for IR casework.

**Phase 0 (1-2 weeks) closes the immediate holes.** CSRF double-submit, `__Host-` cookie prefix, CSP nonce wiring, body-size middleware, `/register` email-enumeration fix, WebSocket Origin check, GitHub Actions pinned by SHA, Dependabot on.

## 2. Context & constraints

### 2.1 Business context

- **AMG** = Anchor Mill Group. Bespoke programs for UHNW individuals, family offices, global executives.
- **Three portals**: internal staff dashboard, partner portal, client portal.
- **IR practice**: AMG offers incident-response services. The platform carries client IR casework (forensics, breach docs, evidence). This elevates *chain-of-custody* and *tamper-evidence* from "nice to have" to product-differentiator status.
- **Stack**: Python 3.12 + FastAPI + SQLAlchemy 2 async + asyncpg + PostgreSQL 17, Next.js 16 + React 19, MinIO, Redis, React Native / Expo, Railway deployment (cross-origin subdomains, `SameSite=none` cookies).

### 2.2 Constraints

- **No paid services.** Everything must be code or free/OSS. Vault Transit, AWS/GCP KMS, Datadog/Sentry paid tiers, Cloudflare Pro, Vanta/Drata SOC 2 tooling, CREST pentest firms — all out.
- **US-only operations for now.** EU/UK/Swiss data residency, SCCs, DPF certification, FCA SMCR, FINMA deferred indefinitely (revisit on first non-US client).
- **Not pursuing broker-dealer licensing.** 17a-4 D3P designation, FINRA WORM evidence — deferred (revisit on affiliation).

- **Internal IR capability.** AMG handles its own security incidents. No Mandiant/Crowdstrike retainer. Internal runbook required.

### 2.3 What we build anyway (even deferred regulatorily)

- **GDPR-style erasure (crypto-shred)** — costs nothing extra to build; makes deferred items near-free to turn on.
- **Retention-friendly architecture (MinIO object lock, tamper-evident audit)** — elevates the product for IR clients today; future-proofs 17a-4.
- **SOC 2-aligned controls matrix** — build the controls, skip the paid audit. Certification is ~$30-80K when needed; the engineering prerequisite is the same regardless.

## 3. Threat model

UHNW + IR-firm has an asymmetric reputational blast radius: the *fact* of a breach is worse than the data lost — names alone reveal wealth association, and IR clients are typically breach victims already.

Ranked by expected harm × likelihood:

1. **Insider / stolen DB credentials.** Today a single leaked `.env` or rogue DBA reads every tax ID, net worth figure, message body, and IR case file in the clear. **Highest-leverage attack, worst outcome, most likely vector.**
2. **Third-party bearer-token leak.** Google/Outlook/DocuSign tokens, DocuSign private key, SMTP password — all plaintext today. A backup leak grants persistent third-party access.
3. **CSRF.** `SameSite=none` cross-origin + no CSRF token = any site the user visits can POST with their cookies. Wide-open.
4. **Presigned URL leakage.** 1-hour TTL, URL-as-token, no audit on fetch. URLs leak into browser history, Referer headers, proxy logs, APM (Sentry/Datadog), stack traces, Slack messages — anywhere a URL can go.
5. **XSS → token exfiltration.** Backend CSP still has `unsafe-inline` for style-src; frontend nonce is generated but unused. Not-if-but-when.
6. **Lost/stolen mobile device.** Biometric gate exists; cert pinning and jailbreak detection do not.
7. **Key compromise.** Single env-var `SECRET_KEY` signs every JWT. No rotation path.
8. **Audit log tampering.** Attacker with DB access rewrites history. No chain verification.
9. **Credential stuffing.** bcrypt (not Argon2id) + no HIBP + IP-only rate limit = distributed stuffing gets through.
10. **Supply chain.** GitHub Actions pinned by tag (mutable). `tj-actions/changed-files` Mar 2025 incident demonstrated this is live exploitation, not theoretical.

## 5. Architectural decisions (locked)

These shape every downstream choice. Changes require explicit re-review.

### D1. Messaging: server-side envelope + supervised break-glass. Not true E2EE.

**Why not E2EE:** Signal/MLS forward secrecy deliberately destroys past keys. An IR firm serving UHNW cannot lose the ability to read historical comms when a case requires replay, a client asks for records, or (if the broker-dealer future ever arrives) a regulator demands supervisory review. True E2EE makes AMG unable to serve its own product.

**Scheme:**
- Per-conversation DEK (AES-256-GCM, 12-byte random nonce, AAD = `conversation_id || sender_id || seq_no`).
- DEK wrapped by tenant-scoped KEK; backend calls unwrap on conversation open.
- DEK rotation every N days or on membership change. Old DEKs retained under "legal hold" flag (not destroyed) so retention + case replay still work.
- **Break-glass access:** 2-person approval (requester + compliance/MD), time-boxed scoped JWT, every decryption emits an audit-chain row with justification text.
- Attachments use per-file DEK with same envelope pattern (see D4).
- Search: keep Postgres `pg_trgm` index on plaintext server-side (server legitimately needs plaintext for supervision); client-side blind indexes add complexity without material benefit.
- In-transit: TLS 1.3 only, WSS, already set up.

**User disclosure:** T&Cs must say "supervised, retained, encrypted at rest, not private from the firm." Different from "E2EE private" — be explicit.

### D2. Key hierarchy: KEK-in-env with versioning + HKDF per-tenant DEKs.

**Day 1 (and probably forever, given free-only constraint):**
- `AMG_KEK_V1`, `AMG_KEK_V2`, ... env vars. 32-byte base64-encoded keys each.
- **Key-id byte in every ciphertext header** selects which KEK to unwrap with. Rotation is lazy (no bulk re-encrypt): add V2, set `CURRENT_KEK_ID=2`, next write uses V2. Old ciphertexts still decrypt via V1 until they're updated.
- **Per-tenant DEK derived via `HKDF(SHA-256, KEK_vN, info=f"amg|tenant|{tenant_uuid}|col|{col_name}".encode())`**. No DEK storage; ~1 µs per derive.
- **Blind-index key is separate** from KEK (`AMG_BIDX_KEY_V1`). Compromise of the index key doesn't compromise plaintext recovery.

**Interface (locked):**

```python
# backend/app/core/crypto.py
class CryptoProvider(Protocol):
    def unwrap_kek(self, key_id: int) -> bytes: ...
    def derive_dek(self, kek: bytes, tenant_id: UUID, column: str) -> bytes: ...
```

`EnvKeyCryptoProvider` is the Day-1 impl. `VaultCryptoProvider` (self-hosted Vault OSS, free) is the future swap — same interface, one line of wiring changes.

**Operational security:**
- Runbook for rotation: generate new KEK, set `AMG_KEK_V{N+1}` in Railway env, deploy, flip `CURRENT_KEK_ID`. Keep N-1 indefinitely for decryption of old data.
- `SECRET_KEY` (JWT signing) gets the same treatment: `SECRET_KEY_V1`, `kid` header in JWTs, multi-key verification window.
- Railway env secrets are the secret store. Tighten Railway project access to strictly named humans.

### D3. Column-level encryption.

**Pattern:** hand-rolled SQLAlchemy `TypeDecorator`. Not `sqlalchemy_utils.EncryptedType` (maintenance mode, `cache_ok=False` breaks SQLA 2 statement caching, single static key, no AAD, no versioning).

**Wire format (every encrypted cell):**

```
[1B version][1B key_id][12B nonce][ciphertext || 16B GCM tag]
```

Stored as `BYTEA`. AAD = `f"{table}|{column}|{pk_uuid}".encode()`. AAD binds the ciphertext to its location — swapping two ciphertexts across rows or columns fails the auth tag verification.

**Encrypt (high-sensitivity fields):**
- `clients`: tax_id, passport, national_id, DOB, net_worth figures, banking details, beneficiary info, KYC document text extracts
- `user`: `google_calendar_token`, `outlook_calendar_token`
- `users/secrets`: DocuSign private key, SMTP password, webhook secrets
- `communications`: message body, email content, voice-memo transcripts
- `programs`: internal notes, financial terms, risk flags
- `documents`: text extracts, OCR output
- `ir_cases` (when built): forensics artifacts, breach notes, evidence metadata

**Do NOT encrypt:**
- Client name, email (needed for login + staff lookup)
- `created_at`, FKs, status, indexed public fields
- Anything you'll query on with `LIKE` / range / prefix

Encrypting indiscriminately kills ORM ergonomics and query performance without adding security.

**Blind indexes** for equality lookup on encrypted fields (tax_id, passport):

```
tax_id_bidx BYTEA  = HMAC_SHA256(BIDX_KEY, normalize(value))[:16]  # B-tree indexed
```

Normalization (lowercase + NFKC + trim whitespace) must be byte-identical on write and on query. Equality only; no prefix, no range. BIDX_KEY rotation requires full recompute — treat as maintenance event.

### D4. File vault: client-side envelope + SSE-S3 + proxy-through + crypto-shred.

**Upload flow:**

```
client --TLS-- FastAPI
                │ 1. validate size + MIME + magic bytes (existing)
                │ 2. ClamAV INSTREAM scan of PLAINTEXT
                │    → quarantine bucket on positive
                │ 3. generate 32-byte DEK
                │ 4. AES-256-GCM encrypt in 1 MiB segments
                │    per-segment 12-byte random nonce
                │    AAD = file_uuid || segment_seq
                │ 5. wrap DEK with tenant KEK
                │ 6. MinIO put_object with metadata:
                │      kek-version, dek-wrapped, nonce-prefix, seg-size, sha256
                │ 7. zeroise DEK from memory (bytearray + overwrite)
                │ 8. DB row: file_id, object_key, kek_version, sha256, clam_result, uploaded_by
                ▼
               MinIO (SSE-S3 on top as defense-in-depth)
               Object Lock: COMPLIANCE mode, configurable retention
```

**Download flow (sensitive files — KYC, contracts, IR casework):**

```
client --GET /files/:id-- FastAPI
                │ 1. row-level authorization decision
                │ 2. audit-chain log the decision
                │ 3. stream object from MinIO
                │ 4. unwrap DEK via CryptoProvider
                │ 5. AES-GCM decrypt segments on the fly
                │ 6. stream plaintext to client; zeroise DEK
                ▼
               client browser (never sees presigned URL)
```

Proxy-through gives **per-byte audit**. Presigned URLs log *issuance* but default MinIO audit logs miss actual fetch — publicly documented on multiple real incidents.

**Download flow (large files, >50 MB):** server issues a **one-time redemption token** to the client; client redeems at `/download/:token` which issues a 60-120s presigned URL. Invalidate token on first use. Records fetch in audit chain.

**Object lock:** COMPLIANCE mode (immutable even by root until retention expiry) for KYC, signed contracts, IR evidence. Default 7 years (industry norm for KYC; IR evidence may need longer based on case).

**Crypto-shred for erasure:**
- Per-client files are bound to a per-client KEK version (e.g. `amg-client-{uuid}-kek-v1` derived via HKDF).
- On legally-processed erasure request: destroy the per-client KEK version + purge `dek_wrapped` from the DB rows. Ciphertext remains on disk (satisfies retention obligation), mathematically unrecoverable (satisfies Art. 17-style erasure). ENISA-recognized reconciliation pattern.

### D5. Audit chain: tamper-evident.

Evidentially admissible output is a product feature for an IR firm.

**Per-row columns (extend existing `audit_log` table):**
- `prev_hash BYTEA` — SHA-256 of the previous row's `row_hash` (zeros for genesis)
- `row_hash BYTEA` — `SHA256(canonical_json(row_minus_hash_cols) || prev_hash)`
- `hmac BYTEA` — `HMAC_SHA256(daily_signing_key, row_hash)` — rotating daily key lives in env under `AUDIT_HMAC_KEY_YYYYMMDD`
- `day_bucket DATE` — indexed; lets you verify one day's chain in isolation

**Daily cron (via APScheduler, which is already in the stack):**
1. Compute Merkle root over all `row_hash` for the UTC day.
2. Sign root with Ed25519 (`cryptography` lib; `ed25519.Ed25519PrivateKey` generated once, stored under `AUDIT_ED25519_PRIVATE_V1` env).
3. Anchor root to **FreeTSA.org** (free RFC 3161 TSA). Fallback: **sigstore** transparency log.
4. Write `(day, merkle_root, signature, tsa_token)` to `audit_checkpoints` table + push to MinIO with object lock.

**Daily verification cron:**
- Walk the day's rows; recompute `row_hash` and `hmac`; compare to stored.
- Recompute Merkle root; verify Ed25519 signature; verify TSA token.
- Alert on mismatch.

**Public-verification asset:** publish the Ed25519 public key on a static URL (e.g. `amg-portal.com/.well-known/audit-key.pem`). Any third party (including AMG's own IR clients) can independently verify the signed roots.

**Role separation:** revoke UPDATE/DELETE on `audit_log` from the app role (`amg_app`). Only `amg_audit_maintainer` (a separate role used by the rotation job + DBAs) can touch it. Application cannot rewrite history even if compromised.

**GDPR erasure against immutable log:** split PII columns (actor_email, subject_name, free-text payloads) into a `pii_jsonb` column encrypted with a per-subject DEK. Erasure = destroy the subject's DEK. Event + chain intact; subject-identifying fields mathematically unreadable.

## 7. Per-domain specifications

### 7.1 Authentication

- **Password hashing:** Argon2id via `argon2-cffi`. OWASP 2026 params: `PasswordHasher(time_cost=2, memory_cost=47104, parallelism=1)`. `check_needs_rehash()` for seamless parameter upgrades. Lazy migration from bcrypt on successful login.
- **MFA:** TOTP (`pyotp`, already in place) with `valid_window=1`. Store `(user_id, last_totp_counter)` and reject replay within the window. Backup codes: 10 codes, single-use, SHA-256 hashed (not bcrypt — high-entropy inputs), regenerate on MFA reset.
- **Passkeys:** `py_webauthn ≥2.7`. `ResidentKeyRequirement.PREFERRED`, `UserVerificationRequirement.REQUIRED`. Platform + cross-platform authenticators both allowed. New `webauthn_credentials` model: `user_id`, `credential_id`, `public_key`, `sign_count`, `transports`, `aaguid`, `last_used_at`. TOTP stays as fallback.
- **Refresh token rotation:** family-based with reuse detection (already implemented). Keep.
- **JWT signing:** migrate to `kid`-headered keys; multi-key verification window; rotate quarterly.
- **HIBP check:** k-anonymity API at password set/change/reset. Free.
- **Account protection:** exponential backoff per `(user_id, ip)` bucket in Redis. Do NOT lock accounts on failures (DoS vector). CAPTCHA (hCaptcha free tier or Cloudflare Turnstile free) after N fails.
- **`/register` fix:** always return 202; differentiate only via email content ("welcome, set password" vs "account already exists, here's a reset link").
- **Step-up auth:** separate short-lived JWT with `purpose=step_up`, `action_scope=["wire_approve"]`, 5 min expiry. Gated ops listed in Phase 2.11.
- **Session idle timeout:** `last_active_at` on `refresh_tokens`; 30 min idle, 7-14 day absolute.
- **Device fingerprint:** hash `UA + IP/24 + Accept-Language` on refresh token issue; mismatch on refresh → force re-auth.

## 9. Blindspots

1. **Background job logs leak PII.** APScheduler/Celery tracebacks land in stdout → Railway logs. Use structured logging with a whitelist formatter; scrub at the logger layer.
2. **Redis holds plaintext sessions + CSRF + rate-limit keys — it's PII.** TLS + AUTH + internal DNS only.
3. **Backups are personal data.** `pg_dump` with no encryption is a full PII export. `pg_dump | age -r <pubkey>` sidecar to R2 with offline-held `age` identity.
4. **WebSocket bypasses HTTP middleware.** WS handler must run the same auth + rate-limit + audit code as REST. Audit every code path.
5. **"Deleted" accounts aren't.** Soft-delete + grace + crypto-shred; true DELETE breaks the audit chain.
6. **Third-party JS on authenticated pages** (analytics, error reporters, chat widgets) can rug-pull. SRI + version pinning; self-host for authed surfaces.
7. **APM / log platforms capture full URLs** — presigned URL tokens land in external systems. Strip `X-Amz-Signature` (and equivalents) at the SDK layer. `Cache-Control: no-store` on signed-URL responses.
8. **RLS pool hazard:** asyncpg connection returned with `SET app.current_tenant` still set = cross-tenant leak. `RESET ALL` on checkin is mandatory; tests must assert.
9. **`hvac` (Vault client) is blocking.** If we ever adopt Vault, every `kms` call stalls the asyncio event loop without `asyncio.to_thread`.
10. **Nonce forces dynamic rendering.** Wiring `x-nonce` in layout makes the tree dynamic; mark with `export const dynamic = "force-dynamic"` where needed.
11. **`Content-Disposition` with user filename** — CRLF/quote injection breaks response split. Use RFC 6266 `filename*=UTF-8''<percent-encoded>`; strip CR/LF.
12. **Demo accounts in `MFA_EXEMPT_EMAILS`** — must not reach production, ever.
13. **IR case files need stronger chain-of-custody** than typical docs. Per-file access log + Merkle-anchored case snapshot is a product differentiator — treat as part of the IR product, not just infrastructure.
14. **Reputational incident comms.** For UHNW/IR breaches, client-first notification before any public disclosure. Pre-draft comms templates.

## 10. Open questions

1. **Chat supervision requirement now vs later.** Phase 2.8 builds break-glass — but what's the UX? MD self-approves? Always requires compliance? Answer shapes the flow.
2. **IR-case retention periods.** Varies per case type / jurisdiction. Default COMPLIANCE-mode 7 years may be short for some cases; indefinite lock is operationally painful. Need policy table.
3. **Railway project access control.** Who has Railway console access (secrets, DB)? Needs documented list + offboarding checklist.
4. **Off-platform encrypted backup target.** Which free-tier S3-compatible (Cloudflare R2 free 10 GB/mo, Backblaze B2 10 GB free, Railway Volume)? Pick one.
5. **`age` identity custody.** Who physically holds the `age` private key for off-platform backup? Two people? Offline device? Split with Shamir's Secret Sharing?
6. **`AMG_BIDX_KEY_V1` rotation posture.** BIDX rotation is a full recompute. Are we OK locking this in for a long run (5+ years) at launch?
7. **Ed25519 audit key publication.** Publish to marketing site `.well-known/audit-key.pem` — what's the rotation plan? Quarterly? Only on compromise? Publish previous keys too so old signatures verify.
8. **Mobile OTA update signing keys custody.** Expo `EAS_UPDATE` signing key — same question as #5.
